# 현황 정합성 점검 프롬프트 (AI에게 "박사님 지시대로 가고 있나?" 물어보는 용도)

작성일: 2026-07-21
용도: 저장소(IEEE_sensors)의 **현재 코드·작업이 박사님이 지시한 방향과 실제로 일치하는지**를
AI(코딩 에이전트)에게 감사(audit)시키는 프롬프트. `SR_평가_프롬프트.md`가 "SR 모델을 고르는"
용도라면, 이 문서는 "지금 가고 있는 게 맞는지 점검"하는 용도다.
사용법: 아래 **[프롬프트]** 블록 전체를 저장소에 접근 가능한 AI에게 그대로 붙여넣는다.
(원하면 §끝의 "빠른 버전"만 써도 된다.)

---

## [프롬프트] — 그대로 복사해서 사용

```
너는 이 저장소(IEEE_sensors)를 처음 인수인계받은 시니어 리뷰어다. 임무는 딱 하나:
"현재 코드와 작업이 지도교수(박사님)가 지시한 방향과 실제로 일치하는가?"를
저장소의 실제 파일을 읽고 근거와 함께 판정하는 것. 코드를 새로 짜거나 고치지 마라.
이건 구현이 아니라 감사(audit)다.

────────────────────────────────────────────────────────
[0] 판정 규칙 (반드시 지켜라)
- 추측 금지. 모든 판정은 네가 직접 읽은 파일의 경로(가능하면 라인/함수명)를 근거로 대라.
- "있을 것이다"가 아니라 "파일 X의 함수 Y에서 확인함" 형태로만 서술하라.
- 근거를 못 찾으면 "미확인"으로 분류하고, 어디를 봐야 확정되는지 적어라.
- 문서(참고자료/*.md)의 주장과 실제 코드가 어긋나면 그 불일치를 반드시 지적하라.
  (예: 핸드오프 문서엔 "단위테스트 통과"라고 적혀 있는데 테스트 파일이 실제로 없다면 지적.)

[1] 먼저 읽을 것 (이 순서로)
- 참고자료/프로젝트_핸드오프.md      : 박사님 방향 최종본 + 완료/남은 일
- 참고자료/STAGE2_결정사항.md        : 승인된 아키텍처·fps·이번 라운드 스코프
- 참고자료/ROADMAP.md, 실험로그.md   : 방향·엣지 실측 근거
- src/eyeblink/config.py             : 상수(캘리브/STD/SR/fps 파라미터)
- src/eyeblink/frontend.py           : 조건부 얼굴 ROI two-pass SR 오케스트레이션 + SrGate
- src/eyeblink/robust.py             : 순수 SR 유틸(cv2.dnn_superres) — 스텁/실동작 여부
- src/eyeblink/landmarks.py          : FaceLandmarker/EAR/w_eye/face_bbox/캘리브
- src/eyeblink/std.py, pipeline.py, metrics.py : STD 상태머신·글루·집계
- src/eyeblink/experiment.py, src/scripts/calibrate.py : 피험자 캘리브 실험
- src/scripts/sr_eval.py, run_live.py, fps_sweep.py : SR 평가·실시간·fps 스윕
- models/ 디렉토리                    : SR 사전학습 .pb 파일이 실제로 있는가?

────────────────────────────────────────────────────────
[2] 박사님이 지시한 "정답"(이것과 대조하라)

(A) 최종 파이프라인 — 8단계가 코드에 이 순서·이 형태로 존재해야 함:
  0. 피험자별 EAR 캘리브(실험 준비)  1. MJPEG 서버 스트림  2. 클라이언트 JPEG 수신
  3. 이미지 품질 평가  →  3.1 개선 필요 시 SR로 얼굴 개선 후 4로 / 3.2 불필요 시 바로 4
  4. MediaPipe 입력  5. 눈 랜드마크 검출  6. EAR 계산  7. STD 업데이트  8. 깜빡임 판단

(B) 우리 method(컨트리뷰션) — 이게 코드에 실제로 구현돼 있어야 논문이 성립:
  method① 피험자별 EAR 캘리브레이션: 개안 n초 → 폐안 → "깜빡이세요" 지시마다 STD가
          1회로 세는지 검증, 피험자당 10회 반복, 전 데이터 CSV 기록(임계값 근거).
  method② 눈 픽셀 품질 기준 "조건부" 경량 SR: 매 프레임이 아니라 눈이 작을 때만(멀거나·
          저해상도) 얼굴 ROI에 SR을 켜는 게이트. Pi5에서 도는 경량 모델.
          ※ 품질 기준은 프레임 전체가 아니라 "얼굴/눈 bbox·눈 가로 픽셀 수"여야 한다.

(C) 이번 라운드 스코프(STAGE2 승인) — 이걸 넘어서면 "이탈":
  - 공개 데이터셋으로 모델 학습·F1 벤치는 "이후 확장"으로 미룸. 이번엔 안 함.
  - 이번 검증은 4인(염한울·정다현·강지민·박덕환) 자체 촬영 + 큐 기반 정답만.
  - SR 평가도 자체 4인 영상을 합성 열화(다운스케일/감마/노이즈)해서 on/off 비교로 완결.
  - 아키텍처: 얼굴 ROI two-pass SR(w_eye=게이트, SR 대상=얼굴 crop, EAR 스케일 불변→역매핑 없음).
  - fps 운영점: 하한 15fps(필수) / 목표 24fps / no-SR ~30fps.

(D) 박사님 "즉시 할 일" 5가지 — 각각 어디까지 됐는지 상태를 매겨라:
  ① 관련 논문 2~3  ② 관련 특허 2~3  ③ Pi5용 경량 SR 모델 찾기
  ④ 실험 설계·평가지표 선정  ⑤ 실험 진행  (+ 파이프라인 그림)

────────────────────────────────────────────────────────
[3] 점검 항목 (각 항목을 준수/부분/미충족/이탈/미확인 으로 판정 + 근거)

  1. 파이프라인 8단계가 코드에 모두 존재하고 순서가 (A)와 같은가? 각 단계의 담당 파일은?
  2. 품질 게이트가 "프레임 전체"가 아니라 "눈 가로 픽셀(w_eye)" 기준인가?
     (landmarks.eye_width_px, frontend.SrGate 확인. 게이트 임계값 SR_W_EYE_MIN은?)
  3. SR이 "조건부"인가 — 매 프레임이 아니라 게이트가 켤 때만 도는가? (frontend.EarFrontend.process)
  4. 아키텍처가 얼굴 ROI two-pass인가? pass1(원본 VIDEO)→게이트→얼굴 crop SR→pass2(IMAGE)→EAR.
     EAR을 crop 좌표계에서 계산하고 역매핑을 생략하는 근거(스케일 불변)가 코드/주석에 있는가?
  5. robust.SuperResolution이 실제로 도는가, 아직 스텁/미검증인가? cv2.dnn_superres 사용?
     models/에 FSRCNN-small_x2.pb(또는 ESPCN) .pb 파일이 실제로 존재하는가? (없으면 sr_eval --sr on 불가)
  6. method① 캘리브가 박사님 프로토콜대로인가 — 개안→폐안→10회 검증, 피험자당 반복, CSV 기록?
     (config.CALIB_*; experiment.py; scripts/calibrate.py; 결과 CSV 스키마 확인)
  7. STD가 4단계 완전 순환(A개안→B감김→C폐안→D뜨기→A) 시에만 1회로 세는가?
     장시간 폐안(졸음/내려봄)을 실격 처리하는가? (std.py, config.STD_MAX_CLOSED_SEC)
  8. fps 운영점(하한15/목표24/no-SR30)이 config에 있고, run_live/fps_sweep로 측정 가능한가?
  9. 이번 라운드 스코프 위반이 없는가 — 공개셋 학습/모델 트레이닝 코드가 "이번 경로"에 끼어들지 않았나?
     (src/_legacy/의 옛 신경망·분류기가 현재 파이프라인에서 분리돼 있는지 확인)
 10. sr_eval.py가 자체 영상 합성 열화(다운스케일/감마/노이즈)를 실제로 구현했는가?
     "큐 기반 정답 정합 + F1/정밀도/재현율, SR off vs on 회복곡선"은 구현됐나 TODO인가?
 11. 로드맵이 필수라고 못박은 "SR go/no-go 파일럿"(SR이 실제로 검출을 돕는지)이 실행·기록됐는가?
     (실험로그.md에 수치가 있나? 없으면 미충족.)
 12. 문서 주장 vs 실제 코드 불일치(단위테스트 존재 여부 포함)를 전부 열거하라.

────────────────────────────────────────────────────────
[4] 출력 형식 (마크다운, 이 순서 그대로)

  ① 한 줄 총평: "대체로 정답대로 감 / 부분 이탈 / 방향 이탈" 중 하나 + 이유 한 줄.
  ② 정합성 표: | 점검항목 | 박사님 기대 | 실제(근거 파일:함수) | 판정 | 비고 |
     ([3]의 12개 항목 전부. 판정은 준수/부분/미충족/이탈/미확인.)
  ③ 박사님 5대 즉시할일(D①~⑤) 진행상태 표: 항목 / 상태(완료·부분·미착수) / 근거.
  ④ "이탈·리스크" 목록: 방향에서 벗어났거나 논문을 위협하는 것(예: SR .pb 부재로 SR-on 실행 불가,
     go/no-go 파일럿 미실행 상태로 SR을 기여로 주장, 단위테스트 부재 등). 심각도(상/중/하) 표기.
  ⑤ "지금 당장 할 일" 우선순위 목록(코드/실험 각각): 무엇을 → 왜(어느 항목을 닫는지) → 산출물.
     사람·하드웨어가 필요해 AI가 대신 못 하는 것(4인 촬영·Pi5 전력/fps 실측)은 따로 표시.
  ⑥ 열린 질문 / 박사님께 확인이 필요한 사항.

절대 하지 말 것: 코드 수정, 추측으로 판정, 없는 것을 있다고 하기, 문서 주장을 근거 없이 신뢰하기.
목표는 "잘돼 있다"고 안심시키는 게 아니라, 방향에서 벗어난 지점을 정확히 집어내는 것이다.
```

## 빠른 버전 (짧게 물어볼 때)

```
이 저장소(IEEE_sensors)에서 아래를 코드 근거로 판정해줘. 추측 금지, 파일:함수로 근거 대기,
코드 수정 금지. 대조 기준 = 참고자료/프로젝트_핸드오프.md + STAGE2_결정사항.md.

1) 파이프라인 8단계(캘리브→MJPEG→수신→품질평가→(조건부 SR)→MediaPipe→랜드마크→EAR→STD→판단)가
   코드에 그 순서로 다 있나? 각 단계 담당 파일은?
2) SR이 "매 프레임"이 아니라 "눈 가로 픽셀(w_eye)이 작을 때만" 켜지는 조건부 게이트인가?
   (frontend.SrGate / landmarks.eye_width_px)
3) 아키텍처가 얼굴 ROI two-pass(pass1 원본→crop SR→pass2 IMAGE→EAR, 역매핑 없음)인가?
4) robust.SuperResolution이 실제로 도나, 스텁인가? models/에 SR .pb 파일이 실제로 있나?
5) 캘리브(개안→폐안→10회 검증, CSV)와 STD 4단계 완전순환+장시간폐안 실격이 구현돼 있나?
6) 이번 라운드 스코프 위반(공개셋 모델 학습 코드가 현재 경로에 끼어듦) 없나?
7) sr_eval의 합성 열화는 됐고, 정답정합·F1·SR on/off 비교는 됐나 TODO인가?
   로드맵 필수인 SR go/no-go 파일럿은 실행·기록됐나?
8) 문서 주장 vs 실제 불일치(단위테스트 존재 여부 등)를 열거해줘.

출력: 한 줄 총평 → 항목별 준수/부분/미충족/이탈 표(근거 포함) → 이탈·리스크 → 지금 할 일 우선순위.
```

## 사용 팁
- 이 프롬프트는 "점검 전용"이다. 나온 이탈·구멍 목록을 보고, 실제 구현이 필요하면
  `SR_평가_프롬프트.md`(STAGE 2)로 넘어가면 된다.
- 판정 결과 중 "미확인"이 많으면, 그 AI가 실제로 파일을 안 읽고 답한 것 — [0][1] 규칙을 다시 강조해 재실행.
- 새 채팅/새 에이전트마다 컨텍스트가 리셋되므로, 이 프롬프트 하나로 매번 "현 위치"를 재점검할 수 있다.
