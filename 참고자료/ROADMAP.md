## 0. 방향 확정 (전사문 반영)

- **논문 1 (가까운 목표):** 라즈베리파이 5에서 **실시간·강건한 깜빡임 검출(횟수/BPM)**.
  MediaPipe FaceLandmarker → EAR → **STD 4단계 상태머신** → 1회 판정. 개인 맞춤 캘리브(EAR 이동평균), Flask MJPEG 서버-클라이언트 구조. 2인 30초 실험 + FPS 스윕.
- **핵심 기여 (논문을 살리는 것):** **엣지용 경량 슈퍼레졸루션(SR) 전처리.** 저조도·저해상도로 랜드마크가 무너지는 환경에서 SR로 프레임 화질을 올려 검출을 강건하게, 그리고 그 SR을 라즈베리파이5에서 **실시간(~5fps)** 으로 구동.
- **폐기/강등:**
  - 옛 신경망·"4단계 분류 모델" → 임의 임계값 껍데기라 **제거**(AI/신경망 뺀 순수 EAR+STD).
  - "완성도/불완전 깜빡임"을 논문 척추로 삼는 것 → **선택/보조**(박사님: 정의가 주관적).
  - 라이브니스/딥페이크 방향 → 폐기.
- 기존 문서 `안구건강_선행사례.md`, `관련연구_및_문제정의.md` = 이제 **배경 자료**.

## 1. 시스템 구성 (논문 Figure)

- **HW/SW 블록 다이어그램:** 카메라(웹캠 / Pi 카메라 / 외부) → **MJPEG 스트리밍 서버(Flask)** → **Pi5 클라이언트**(프레임 수신) → [SR 전처리] → FaceLandmarker(눈 키포인트) → EAR → **STD decision** → 횟수/BPM.
  - 서버-클라이언트 구조라 카메라 위치와 무관하게 이식성↑(웹캠도, Pi 직결도, 외부도 동일).
- **STD 다이어그램:** `A(개안) → B(감기) → C(폐안) → D(뜨기) → A` 전체 순환 시 1회 판정. 실눈·프레임 노이즈 오검출 방지. 4단계 운동학 정의는 선행문헌 인용(닫힘/폐안/열림 속도).
- **SR 삽입 위치:** 클라이언트 수신 프레임 → (저조도·저해상도 감지 시) SR 전처리 → FaceLandmarker 입력.

## 2. 선행조사 결과 (이번에 조사)

### (A) 엣지 SR — 신규성·실현성 점검

- ⚠️ **"SR로 저해상도 얼굴 랜드마크 개선"은 이미 포화 영역.** Super-FAN(2018), EfficientSRFace(2023), low-res 얼굴 랜드마크(2019) 등 다수. → *순수 SR-랜드마크*만으로는 신규성 약함.
- **우리 신규성 = 조합.** "엣지(Pi5) + 실시간 + 깜빡임 검출 응용 + fps·전력 실측"으로 차별화해야 함. 관련연구는 계속 파야 함(박사님도 지적).
- ⚠️ **주의(방법론 리스크):** 순차(SR→검출)가 결합(joint)보다 못하다는 결과, "SR 영상이 학습분포와 달라 오히려 검출이 나빠지는" 사례 존재. → **SR이 실제로 검출을 돕는지 초기 검증 필수.**
- **실현성:** 경량 SR(ESPCN ~21k params, FSRCNN, edge-SR)은 Pi CPU에서 **2×/3×는 부분 성공, 4×·고해상도·고fps는 어려움.** TFLite/NCNN + INT8 양자화로 최적화. → ~5fps 목표는 소규모 배율(2×)에서 plausible하나 **Pi5 실측 필요(최대 리스크).**

### (B) 데이터셋 — IRB 없이 개발·검증 가능

- **깜빡임 영상 공개셋:** Eyeblink8(4인, 408 blinks, 640×480, 머리움직임·표정 포함), ZJU(20인, 정면/위/안경/비안경, 30fps, 320×240), RT-BENE(ICCVW'19), **mEBAL2(38인, RGB+NIR — 저조도용으로 유용)**, CEW/MRL Eye, TalkingFace.
- **졸음(프록시):** NTHU-DDD, YawDD (서서히 감기는 눈 영상).
- **활용:** (1) 검출기 개발·검증을 IRB 없이 수행, (2) 공개셋을 **다운스케일/암전**해 저해상도·저조도 테스트셋 생성 → **SR 유무 비교 실험**의 재료. (ZJU 320×240은 이미 저해상도라 그대로도 유용, mEBAL2 NIR은 저조도 시나리오.)

## 3. 작업 순서 (Phase)

- **Phase A — HW/환경** (다현이 담당 가능): Pi5 OS 굽기(Raspberry Pi Imager에 연구실 WiFi SSID/PW 입력) → 헤드리스 → 공유기 DHCP에서 IP 확인 → SSH/VNC. 카메라 Logitech StreamCam C980(1080p/60fps, USB-C → 허브/젠더) 확보. 젖은 손·핀 쇼트 주의.
- **Phase B — 파이프라인 코드:** Flask MJPEG 서버 + Pi 클라이언트 + FaceLandmarker + EAR + **STD 4단계 상태머신** + 이동평균 캘리브. (기존 `detector.py`·`blink_segmenter.py` 재활용, 신경망 제거)
- **Phase C — 실험1(기본):** 2인 30초 촬영, 보조자 큐(3·6·9초…)로 정답 기록 → **인식률** + **FPS 스윕(60/30/15)** 으로 "15fps로도 충분한가" 검증.
- **Phase D — SR 실험(기여):** 공개셋 다운스케일/암전 → SR 유무 검출 정확도 회복 + Pi5 fps/전력. 경량 SR 후보(ESPCN/FSRCNN/edge-SR) 비교, INT8/TFLite/NCNN.
- **Phase E — 초안:** 4p Sensors Letters. 구조는 Carcellar 논문처럼: 문제·필요성 → 시스템(그림 2종) → 실험1 → SR 기여 → 결론.

## 5. 리스크 / 확인 필요

- **SR이 실제로 검출을 돕는가** — 순차 SR이 오히려 손해라는 사례 있음 → 초기 검증.
- **Pi5에서 경량 SR 실시간(~5fps) 실측** — 되는 배율(2×)·모델 확인.
- **SR-on-edge 신규성** — 응용 조합으로 차별화, 관련연구 지속 조사.
- **IRB 필요 여부** 확인(일단 2인 내부 실험으로 완성 우선).
- **깜빡임 지속 ~0.5초** 근거(600fps 측정 시 ~0.57초) → 저fps로도 충분함을 FPS 스윕으로 입증.
