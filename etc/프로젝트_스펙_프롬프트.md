# 프로젝트 스펙 / 작업 지시 프롬프트

> 이 문서는 작업 중인 모델·코드에 대한 **전체 스펙 겸 작업 지시서**입니다.
> 교수님 피드백을 반영해, 무엇을 만들고 무엇을 비교·검증해야 하는지를 정리했습니다.

---

## 1. 프로젝트 개요

- **주제:** 눈 피로 / 눈 깜빡임 감지 (눈 건강 중심)
- **엣지 디바이스:** 라즈베리파이에서 **추론(inference)만** 수행
- **학습(training):** 외부 PC/서버에서 수행 → 학습된 모델을 라즈베리파이로 배포
- **투고 목표:** **IEEE Sensors Letters** (짧은 레터 형식)
- **핵심 메시지:** *"모델을 이만큼 경량화했지만, 성능(정확도) 수치는 별반 다르지 않다."*

---

## 2. 투고 전략 (논문의 핵심 주장)

이 연구의 셀링포인트는 **경량화 대비 성능 유지**다. 따라서 실험은 아래를 명확히 증명해야 한다.

1. **정확도/성능은 거의 그대로 유지** — 경량화 전(baseline) vs 후(lightweight) 성능 차이가 미미함
2. **모델 크기·파라미터는 크게 감소** — 경량화의 실효를 수치로 제시
3. (보조) **라즈베리파이 추론 속도/지연 개선** — 엣지에서 실사용 가능함을 뒷받침
4. (보조) **메모리/전력 사용량 감소** — 엣지 적합성 근거

> 우선순위: **1·2를 메인 주장**, 3·4를 이를 뒷받침하는 근거로 배치.

> ⚠️ **중요 (아래 2-1 참고):** 위 경량화·성능 주장은 반드시 **센서 시스템에 대한 기여**로 연결해서 서술해야 함. "정확도 유지하며 경량화" 자체만으로는 IEEE Sensors Letters에서 **out of scope로 리뷰 없이 반려**됨.

---

## 2-1. IEEE Sensors Letters 투고 요건 (필수 반영)

### (A) Scope — 센서 관련성이 없으면 반려
- **저널 범위:** 물리·화학·생물 현상을 감지/변환하는 **디바이스(센서)** 의 이론·설계·제작·응용. 전자·물리적 측면 강조.
- **AI/ML 범위 규정 (가장 중요):** *AI/ML 알고리즘·데이터 분석·신호처리만 다루고 센서 디바이스/시스템에 대한 **측정 가능한 영향**이 없으면 out of scope → 리뷰 없이 declined.*
- **반드시 보여줘야 할 것:** AI/ML이 **센싱 기술 발전**에 어떻게 기여하는지 —
  - 디바이스 성능 향상 / 시스템 통합 / **센서 특화 적응(edge deployment, noise resilience 등)**

### (B) 이 논문의 프레이밍 전략
이 연구를 **"ML 경량화 논문"이 아니라 "엣지 센서 시스템 논문"으로** 서술한다.
- **센서:** 눈 피로/깜빡임을 감지하는 **카메라(이미지 센서)** 기반 센싱 시스템
- **핵심 스토리:** 모델 경량화 → **라즈베리파이 엣지에서 온-디바이스 추론이 가능해짐(edge deployment)** → 클라우드 전송 없이 **센서 노드 자체에서 실시간 눈 건강 센싱** 실현
- **측정 가능한 센서 시스템 기여로 연결:**
  - 온-디바이스 추론 지연/처리량(FPS) → **실시간 센싱 성능**
  - 메모리/전력 절감 → **센서 노드의 자립성·상시 구동 가능성**
  - (가능하면) noise resilience, 조명 변화 등 **실제 센싱 환경 강건성**
- Abstract·Introduction·Conclusion에서 **"sensor system / edge sensor node / sensing performance"** 언어를 명시적으로 사용. 알고리즘 자체가 아니라 **센서 시스템의 성능·활용성 개선**이 주장의 중심이 되도록.

### (C) 형식·제출 규정
- **분량: 최대 4페이지** (참고문헌 최소 1단(single column) 포함) — 초과 시 IEEE Sensors Journal로 유도됨. 짧은 레터이므로 군더더기 없이 압축.
- **학회 논문 재출판 불가:** 이미 conference proceedings에 실린 내용은 게재 불가.
- **제출:** IEEE Author Portal을 통해 전자 제출만. EIC/AE에게 직접 보내지 말 것.

## 3. 시스템 아키텍처

```
[외부 PC/서버]                         [라즈베리파이]
  모델 학습(training)                    추론(inference)만
      │                                     │
      └── 학습 완료 모델 ── ONNX 변환 ──▶ ONNX 런타임으로 추론
                                            │
                                            └── 결과 전송 (현재 Flask 사용)
```

- **학습/추론 분리:** 학습은 반드시 외부에서, 라즈베리파이는 추론 전용
- **모델 포맷:** 학습된 모델을 **ONNX로 변환**하여 라즈베리파이에서 경량 런타임(onnxruntime)으로 구동 → 프레임워크 종속성 제거 + 추론 최적화
- **통신:**
  - **현재:** Flask 기반 통신 사용 중 (유지)
  - **참고(선택):** Pyro(Python Remote Objects) 같은 파이썬 원격 통신 라이브러리도 대안으로 검토 가능 — 교수님이 "이런 것도 있다" 정도로 언급하신 참고 사항이므로 **필수는 아님**. 지금은 Flask 유지, Pyro는 향후 옵션으로만 기록.

## 4. 실험 · 비교 설계 (경량화 전후)

아래 표 형태로 baseline vs lightweight를 비교 제시할 것.

| 지표 | Baseline (경량화 전) | Lightweight (경량화 후) | 목적 |
|---|---|---|---|
| **정확도 / F1 등 성능** | (측정) | (측정) | **거의 동일함**을 강조 (메인) |
| **파라미터 수 / 모델 크기(MB)** | (측정) | (측정) | **큰 폭 감소**를 강조 (메인) |
| **추론 지연(latency) / FPS** (라즈베리파이) | (측정) | (측정) | 엣지 실사용성 (보조) |
| **메모리(RAM) / 전력** (라즈베리파이) | (측정) | (측정) | 엣지 적합성 (보조) |

- 추론 속도·메모리·전력은 **반드시 라즈베리파이 실기기 기준**으로 측정
- 정확도는 동일 테스트셋으로 공정 비교
- 경량화 기법(pruning/quantization/경량 아키텍처 등)을 명확히 기술
